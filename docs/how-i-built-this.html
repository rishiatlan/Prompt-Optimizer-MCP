The Problem I Kept Running Into

I use AI constantly — for writing, coding, planning, research, emails. Claude is my daily driver. And I kept noticing the same pattern:

People (including me) are terrible at telling AI what they actually want.

“Make the code better.” “Fix the login bug.” “Write me a blog post.” These sound reasonable. But they’re the AI equivalent of walking into a restaurant and saying “give me food.” You’ll get something, but probably not what you wanted.

The result? Wasted iterations. Wasted tokens (which cost money). Wasted time going back and forth with the AI trying to clarify what you meant in the first place.

TL;DR
Vague prompts lead to vague results, which lead to wasted money and time. Everyone knows this, but nobody systematically fixes their prompts before hitting Enter.

The “Aha” Moment

I was reading about Model Context Protocol (MCP) — a new open standard from Anthropic that lets you extend Claude with custom tools. Think of it like plugins for AI: you can give Claude the ability to search databases, control your browser, call APIs, or — in my case — analyze and fix prompts before they execute.

The idea hit me: what if there was a tool that sat between you and Claude, intercepting your sloppy prompt, scoring it, restructuring it, and compiling it into a professional instruction set — all before Claude even sees it?

Like a spellchecker, but for AI prompts. And it would do this with zero AI calls — pure rules and algorithms. That way it’s instant, free to run, and deterministic (same input always gives same output).

What Is an MCP, and Why Should You Care?

Quick explainer for the non-technical crowd:

MCP (Model Context Protocol) is a standard that lets you give AI assistants new abilities. Normally, Claude can only chat with you. With MCP, you can give it “tools” — specific capabilities that it can call during a conversation.

My MCP gives Claude 20 tools, grouped into four categories:

  • Governance: Optimize prompts, enforce policies, score risk, classify task complexity, route to optimal models
  • Operations: Compress context, prune irrelevant tools, run full pre-flight analysis before any work begins
  • Enterprise: Configure policy modes, lock config with passphrases, manage sessions, export hash-chained audit trails
  • Utility: Estimate costs across 4 providers, check prompt quality, manage licenses, view usage stats

When you install the MCP, Claude can automatically use these tools. You type a vague prompt, Claude calls the governance engine, and you get back a structured, policy-compliant prompt before any work begins.

TL;DR
An MCP is a plugin system for AI. Mine adds prompt governance. Install it once, and Claude enforces quality and policy on every prompt.

Designing the Product

Core Principles

I set four non-negotiable rules before writing a single line of code:

  1. Zero AI calls inside. The optimizer itself makes no API calls. All intelligence comes from rules, patterns, and scoring algorithms. This means it’s instant, free to operate, and never produces hallucinated advice.
  2. Works for any LLM. The same prompt compiles to different formats: Claude gets XML tags (<role>, <goal>, <constraints>), OpenAI gets system/user splits, and generic targets get Markdown headers. One prompt, three outputs.
  3. Nothing ships without user approval. The MCP never silently finalizes a prompt. If your input is ambiguous, it asks blocking questions first. You review the compiled result, see the quality score and cost estimate, and explicitly approve before anything executes. approve_prompt hard-fails if unanswered questions remain — the gate is enforced in code, not convention.
  4. Four tiers from day one. Free users get 10 lifetime optimizations — enough to see the value. Pro and Power unlock higher volume. Enterprise adds policy enforcement, audit trails, and config governance.

The Pipeline

Every prompt goes through a 5-stage pipeline:

Pipeline
Your prompt → ANALYZE → SCORE → COMPILE → CHECKLIST → COST ESTIMATE
  1. Analyze — What does the user actually want? Is this a coding task, a writing task, research, planning? What are the implicit assumptions? Are there blocking questions?
  2. Score — Rate the prompt 0–100 across multiple quality dimensions, each with weighted penalties for specific weaknesses
  3. Compile — Restructure it with a role definition, success criteria, safety constraints, workflow steps, and uncertainty policies
  4. Checklist — 9-point structural coverage check (does it have a goal? constraints? inputs? outputs?)
  5. Cost Estimate — Exact token counts and dollar costs across 11 models from 4 providers

And then the approval gate: the user always reviews the compiled prompt before it’s used. If the MCP detects ambiguity, it surfaces up to 3 blocking questions that must be answered before the prompt can be approved. approve_prompt hard-fails if unanswered questions remain — no prompt ships without explicit user sign-off.

Average improvement: +32 points. A vague prompt scoring 48/100 comes out at 90/100 after compilation.

TL;DR
Five deterministic stages turn any prompt from amateur to professional, with a mandatory human approval gate at the end. No AI used in the process — pure pattern matching and rules.

Building It — With AI as My Co-Pilot

Here’s the irony: I built a prompt governance engine using the AI it’s designed to help.

The entire project was built with Claude Code — Anthropic’s AI coding agent. I wrote the architecture, designed the interfaces, and specified what each module should do. Claude wrote the implementation, tests, and documentation.

The Stack

  • TypeScript — type safety for a rules engine is non-negotiable
  • MCP SDK — the official Anthropic library for building MCP servers
  • Zod — runtime schema validation for all tool inputs
  • Node.js crypto — Ed25519 signatures for license keys (zero external dependencies)
  • No database. Local file storage in ~/.prompt-control-plane/

That’s it. Two runtime dependencies: the MCP SDK and Zod. Everything else is Node.js built-ins.

The Hardest Problems to Solve

1

Detecting What People Actually Mean

“Write a post about my MCP” — is that a coding task or a writing task? The word “post” could mean a blog post, a social media post, or a POST API endpoint.

I built a three-layer detection system:

  • First, look for output type keywords (“post”, “article”, “tweet” → writing)
  • Then, run full-prompt pattern matching for code/debug/refactor signals
  • Finally, fallback heuristics that catch edge cases

The intent-first approach handles conversational prompts that keyword matchers miss.

2

Scoring Without Being Arbitrary

It would be easy to slap a score on a prompt and call it a day. But people would immediately ask “why did I get a 48?” and you’d better have a real answer.

Each score breaks down into multiple dimensions with clear penalties. Every point deducted has a specific, traceable reason — vagueness, missing constraints, scope issues. The math is transparent and reproducible.

3

Offline License Validation

Most products phone home to check if you’ve paid. I didn’t want that. My users might be on planes, behind corporate VPNs, or philosophically opposed to telemetry.

Solution: Ed25519 cryptographic signatures. I sign each license key with a private key (kept secret). The optimizer verifies the signature using only the public key (embedded in the code). No internet required. No server to maintain. No external dependency.

The trade-off: if I ever lose the private key, I can’t generate new licenses. And if I regenerate the key pair, all existing licenses become invalid. High stakes, but clean design.

4

The Freemium Gate

How do you meter usage fairly without a server? Local file storage tracks usage counts, but what stops someone from deleting the file?

Answer: nothing. And that’s fine. The free tier is a funnel, not a prison. If someone wants to hack around the limit, they’re technical enough to have opinions about the product. The license system handles the actual paywall for Pro/Power features.

TL;DR
Intent detection, transparent scoring, offline licensing, and honest freemium design. Each one was harder than it looks.

Hiccups, Headaches, and Hard Lessons

The npm Token Saga

Every. Single. Time. I tried to publish to npm, the auth token would fail with a cryptic “one-time password required” error. The issue? npm has three types of tokens — Publish (needs OTP), Read-only, and Automation (no OTP). I kept accidentally using the wrong one across sessions.

Lesson learned: I documented the exact token type and recovery steps in the project docs so no future session makes the same mistake.

The Key Rotation Emergency

I regenerated my Ed25519 key pair during development... forgetting that I’d already generated test license keys with the old pair. Suddenly no licenses validated. The private key signs, the public key verifies, and they must match. New key pair = all old keys are dead.

Lesson learned: Key management is a one-way door. Document it, back it up, and never regenerate casually.

Smithery’s ESM Incompatibility

Smithery (one of the major MCP directories) uses an internal bundler that forces all code into an older JavaScript format (CommonJS). My project uses the modern format (ESM) with features like top-level await that physically can’t run in CJS. Build error. No workaround.

Lesson learned: Not every distribution channel will support your tech choices. Ship what you can, document the rest.

The MCP Registry’s mcpName Requirement

The Official MCP Registry requires a special field (mcpName) in your npm package. But you have to publish to npm first with that field, then publish to the registry. I discovered this after multiple failed attempts. Had to bump to v2.2.2 just to add one metadata field.

Lesson learned: Read the entire docs before starting a submission pipeline. Version bumps are cheap; frustration is expensive.

Cloudflare Pages Deployment: Git Push ≠ Live

Migrated hosting from GitHub Pages to Cloudflare Pages for v4.0.2. GitHub Pages auto-deploys on push; Cloudflare Pages does not. Every change to docs/ requires a manual wrangler pages deploy docs/ --project-name prompt-control-plane step. I learned this the hard way after pushing docs updates and waiting 30 minutes wondering why the live site hadn’t changed.

Lesson learned: Multi-step deployments need explicit handoff steps. Document them, add them to CI, or automate them away. Git push is a dangerous illusion of "done"—especially when your docs site is the product’s only interface to the world.

Multi-Release Version Sync Across 15+ Surfaces

Shipped five major releases (v4.0 → v4.1.0) within five days. Each release touched npm (package.json, package-lock.json), GitHub (tags, releases), docs site (11 pages, each with version mentions), README, CHANGELOG, schema docs, and feature matrices. Tool count went 4 → 19 → 20. Keeping version numbers, feature lists, and tier descriptions in sync across all surfaces manually was error-prone. Almost shipped v4.0.2 docs claiming 19 tools when we had 18.

Lesson learned: Centralize versioning in a single source (one JSON file), then have a build step propagate it everywhere: README, docs site, API references, and changelogs. Never manually edit version numbers in 15 places. Reduce cognitive load; reduce mistakes. The version is the single source of truth—everything else derives from it.

Content Filtering Across 15+ Public Surfaces

For v4.1.0, I stripped internal references (source paths, function names, type names, test counts) from README, CHANGELOG, docs site (11 pages), and API reference to keep public documentation business-focused. The challenge: ensuring consistency across all surfaces. I found outdated paths in one doc, exposed function names in another, and test counts scattered across three pages. Fixing them manually was tedious and error-prone. Almost shipped with internal jargon mixed into marketing copy.

Lesson learned: Content filters for "public vs internal" should be applied early in the docs pipeline, not retroactively. Consider separate source files for internal vs public content, or use build-time templating to strip sensitive details. One missing filter in one file defeats the entire exercise.
TL;DR
npm tokens are confusing, key rotation is irreversible, not all directories support modern JavaScript, always RTFM before registries, git push doesn’t mean live on Cloudflare Pages, version numbers in 15 places is a recipe for inconsistency, and content filtering across 15 surfaces is error-prone unless templated. Centralize, automate, and template.

Update: v4.0.0 and the Evolution to Governance

Since the initial 49-hour build, the product went through a rapid evolution across multiple releases. Each one expanded the surface area without adding a single LLM call. Here is the arc through v4:

v3.0: The Decision Engine

Added a pre-LLM decision layer. Before a prompt even gets optimized, the engine now classifies its reasoning complexity (6 types), suggests an optimization profile (5 presets), and routes it to the optimal model across 4 providers and 11 models. Every routing decision produces a full decision_path audit trail showing exactly why a model was chosen. Risk scoring with dimensional weights (hallucination, ambiguity, scope, constraint) drives the routing logic. All still deterministic.

v3.1: Smart Compression + Tool Pruning

Built a 5-stage heuristic compression pipeline (H1 through H5) that strips license headers, collapses consecutive comments, deduplicates identical lines, stubs empty function bodies, and truncates middle sections when aggressive mode is on. A zone scanner protects structured content (code blocks, tables, lists, JSON, YAML) from being mangled. Added a tool pruning engine that scores MCP tools by relevance to the current task and marks low-value tools for removal, saving context tokens. Four new ambiguity detection rules brought the total to 14.

v3.2: Enterprise Tier + Distribution

Introduced the Enterprise tier with 120 requests/min and always_on mode. Split the monolithic landing page into a multi-page docs site with dark/light theme. Shipped a CLI binary (originally prompt-lint, later expanded to pcp with 11 subcommands in v5) so teams can lint prompts in CI without MCP. Published a GitHub Action so any repo can add prompt governance to pull request checks. Moved hosting from GitHub Pages to Cloudflare Pages for faster global edge delivery.

v3.3: Enterprise Operations

The release that made it a real governance product. Policy enforcement with advisory and enforce modes, where enforce actually blocks deployment on violations. A hash-chained audit trail using SHA-256 hash chaining (if any entry is modified, all subsequent hashes break). Config lock with passphrase protection so governance settings cannot be changed without authorization. Session lifecycle management with delete, purge, retention policies, and dry-run safety. Custom rules engine where teams define their own detection patterns with regex, severity levels, and risk dimensions.

v4.0: Prompt Control Plane

Renamed the product. Updated every surface: npm, GitHub, docs site, directories. The rebrand reflected what the product had become. Not a linter. Not an optimizer. A control plane for how organizations manage AI prompts. Enterprise tier gates enforced in code. Reproducible session exports with auto-calculated rule-set hashes. Professional documentation site at getpcp.site.

v4.0.2: Cloudflare Pages Migration

Migrated hosting from GitHub Pages to Cloudflare Pages for global edge delivery and enhanced security. Added custom headers, redirect rules, and a 404 page. Expanded website from 5 pages to 11+ pages, including a dedicated blog, privacy policy, terms of service, and security documentation. Shipped trust signals: Shields.io badges for version, license, and security status. Three-column footer with product, resources, and legal links across all pages. Zero downtime migration completed within 24 hours of v4.0 launch.

v4.0.3: Privacy & Contact Infrastructure

Replaced all exposed email addresses in documentation, legal pages, and footer with a general contact form (privacy-first, no SMTP exposure). Added dedicated "Contact" link to footer across all pages. Broadened the contact form from enterprise-only to general audience, with a subject dropdown (General, Support, Enterprise). Implemented post-submission confirmation page and email integration via Formspree. Security-hardened legal pages (privacy.html, terms.html, security.html) to remove internal references and operational details.

v4.1.0: Enterprise Console Integration

Shipped Tool 20 — save_custom_rules (March 3, 2026). Enterprises can now build custom governance rules in the Enterprise Console (admin panel), then click "Copy & Deploy" to paste directly into any MCP-connected AI assistant. Rules are validated, sorted deterministically, written with secure file permissions (chmod 600), and return a rule-set hash for reproducibility. Works with any LLM — not Claude-specific. Enterprise Console provides a visual rule editor, policy enforcement dashboard, audit trail viewer, and role-based governance controls. Consolidated all Enterprise Features (Policy Enforcement, Config Lock, Audit Trails, Custom Rules, Session Management) into a dedicated section on the website. Updated terminology across all 15+ public docs: "tamper-evident" → "hash-chained" to describe the integrity verification mechanism. Content filter applied: stripped internal source paths, function names, type names, test counts, and implementation details from README, CHANGELOG, and all website pages to keep the public documentation focused on business value, not implementation.

All --json output uses a consistent envelope: request_id (UUID v4), version, schema_version, and policy_mode. When governance blocks execution in enforce mode, the CLI exits with code 3 and includes violation details in the JSON output.

Auto-Check Hooks

The hook subcommand lets developers automatically check every prompt before it reaches the LLM:

Terminal
pcp hook install # project-level pcp hook install --global # all projects pcp hook install --threshold 75 pcp hook status pcp hook uninstall

Works with any MCP client that supports UserPromptSubmit hooks (Claude Code, Cursor, Windsurf). The hook runs pcp check on every prompt and warns if the score falls below the threshold.
